## 前言

最近几周一直遇到一个 OGG 问题，有一张表已更新就会中断 OGG 同步进程，本文记录一下分析过程以及解决方案。

## 问题描述

昨天下午，客户说 OGG 又停了（OGG 版本是 19.1.0.4，源端和目标端同版本，都是 19.20）：

> ogg 又停了，源端到目标端的 XZ3WFP 链路，可以查一下原因吗？

连上环境看了下，确实目标端复制链路处理 ABENDED 状态：

![OGG 更新表频繁导致进程中断，见鬼了？非也！](/images/9486c7a71e91dae383212dc7931f7ad0.png)

查看目标端 ggserr.log 日志（日志路径一般在 $OGG\_HOME 目录下）：

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图2](/images/57d4f10c575435b48896c7e1573b2e50.png)

日志里没有看到明显报错信息。

## 问题分析

简单查看了下每次链路 ABENDED，报错的内容都是因为 update 语句导致，源端都是正常抽取状态，目标端复制进程异常终止。

关于 ggserr.log 日志中报错信息，查看官方针对这个错误代码的定义：

```
OGG-01296: Error mapping from {0} to {1}
The mapping of the specified source and target tables failed.

Examine the accompanying messages that provide details about the mapping failure, and resolve the problem based on those messages.
```

查看目标端 discardfile 文件（日志路径一般在 $OGG\_HOME/dirrpt 目录下）：

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图3](/images/65557a17014707752799ba6901abba36.png)

看报错是缺列 ID，ID 是这张表的主键，检查了一下源端和目标端的表结构以及主键，发现没有区别，两端保持一致：

```
SQL> select owner, constraint_name, constraint_type, status, validated from dba_constraints where owner='XXXXXXX' and TABLE_NAME = 'XXXXX';
```

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图4](/images/ec30598f11b9900c7885d5762248ae04.png)

源端和目标端均存在主键，按理来说应该没有问题。问了一下客户，这张表是最近新建的表，检查一下源端的抽取链路参数：

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图5](/images/3c2ad571818ea0875bc004a28ba92df0.png)

有设置参数 `ddloptions addtrandata report` 参数，按理来说应该可以自动添加 trandata，检查一下源端 trandata 是否正常（一般是手动添加完 trandata 用 dba\_log\_groups 视图查看）：

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图6](/images/573f329090f7f2cfb4ea087da09f8621.webp)

但是通过参数自动添加需要使用下方方式查看：

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图7](/images/2da8e78ca525c923a5052f0885791f15.webp)

检查发现没有添加 trandata 记录，在 ggsci 中检查表是否添加 trandata：

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图8](/images/7794f018a61b5b20efe22e44d813228b.webp)

确实没有添加 trandata，这就很奇怪，不过先处理问题要紧。

## 解决方案

之前每次遇到这个问题，为了紧急恢复同步，都是使用跳过当前事务的方式恢复：

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图9](/images/540ec58b27fcf655b8962911b301501e.webp)

这里也是先紧急修复了，然后手动添加 trandata：

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图10](/images/8b45c4eb17d0134cea36103eed4d09eb.webp)

添加完之后再次检查视图：

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图11](/images/81f8b2a17a0adc37e308631d91785daf.webp)

可以查看记录，这次是添加成功了，后续请客户再次执行 update 操作：

![OGG 更新表频繁导致进程中断，见鬼了？非也！ - 图12](/images/e5d96335ab5c7d40110d8af8bcb97fbb.png)

复制链路没有中断，数据正常同步，问题解决。

## 后续

后续问了客户建表的操作，客户反馈：

1. 第一次建表忘记添加主键，所以添加成功后，删掉了这张表；
2. 重建表，设置 ID 为主键。

之前每次对这张表进行 update 操作，都会导致复制链路中断。

等后面有时间了可以做一下测试~
